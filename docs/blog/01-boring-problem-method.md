# The Boring Problem Method

- **Status:** outlined
- **Type:** philosophy / flagship (opinion + build-in-public)
- **Serves:** thought leadership + personal
- **Sources:** AutoSpendTracker, Hermès monitor, KinoWeek, BoringHannover
- **Length:** ~900 words
- **Why first:** needs zero research — you've lived it. Locks the voice.

## Hook

The best side projects aren't ambitious. They're *annoying*. The ones I
actually finish all start the same way: something in my day irritated me
enough to spend a weekend killing it.

## Argument

Annoyance-driven beats ambition-driven. A small, specific irritation gives
you a razor-sharp spec, an obvious "done," and a user (you) who'll actually
use it tomorrow. Grand side projects die in the second week; boring ones ship.

## The recipe (name it)

Small personal annoyance → Python + an LLM + Telegram → shipped in a weekend.
Same three ingredients every time. Boring on purpose.

## Three of my own

- **AutoSpendTracker** — money leaking across MXN / EUR / USD after moving to
  Mexico. Gmail + Gemini + Sheets. The annoyance: "where did it go?"
- **Hermès monitor** — a stock-check I was doing by hand. Scrape + Vision +
  Telegram alert. The annoyance: refreshing a page.
- **KinoWeek** — "is anything on this week in Hannover?" A stateless weekly
  digest. The annoyance: 12 tabs every Friday.

## Takeaway

Keep a running list of things that annoyed you this week. That list is your
best project backlog — better than any "app ideas" note, because every item
already has a user who wants it.
